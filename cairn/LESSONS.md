# Lessons

Durable repo lessons — build quirks, testing tricks, gotchas worth
remembering next time — captured at milestone end and surfaced at plan time.
Not status, not decisions: a lesson is a reusable "how this repo actually
behaves" note. Cross-cutting *choices* still go to `DECISIONS.md`.

One line per lesson: `- YYYY-MM-DD (M<NN>): <lesson>`. Two caps, two remedies
(tracking-rules weight-caps): 50 lines — prune the stalest when full — and
17,000 characters, where the fix is compressing or consolidating in place, not
evicting. Current knowledge: a lesson proven false is corrected in place (D-045).

- 2026-07-11 (M16): this repo has no CI — `gh pr checks --watch` returns "no checks reported" + exit 0; treat as no-CI and merge directly, never wait for green.
- 2026-07-12 (M32): this repo has no `pytest` — the script suite is unittest; run `python3 -m unittest discover -s scripts/tests` (or `-k` a name via `python3 -m unittest scripts.tests.test_scripts -k <sub>`). Don't write `pytest …` into acceptance criteria; it fails "No module named pytest".
- 2026-07-12 (M35): when a milestone's scope was drawn from existing ROADMAP candidate rows, those rows are NOT pruned at plan time (candidates graduate at completion, not at plan) — graduate them in the post-merge hygiene pass or the ROADMAP advertises shipped work as pending; both review lenses flagged M35's two fulfilled rows.
- 2026-07-12 (M51): the archive summary has its own tight ≤25-line cap (separate from the 150-line milestone cap) — a decisions-heavy milestone won't fit a full Goal/Outcome/Decisions/Review skeleton, so budget ~2 wrapped lines per section. Trimming prose only reduces the count when it removes a whole *wrapped* physical line: a shorter sentence that still wraps to N lines saves nothing, so `wc -l` after each edit rather than eyeballing word count.
- 2026-07-13 (M53, extended M54): the false-coverage guard trap (M39/M40) is mechanized — prose-guards register in `skills/tests/test_mutation_harness.py`, which blanks each registered block and asserts its guard fails; a completeness meta-test fails CI on an unregistered guard *file*. Registration is per file (≥1 exemplar block), NOT per assertion — a new `assertIn` in a registered file still needs its own `Mutation(...)` entry or the by-hand M47 check; M23/M26 still apply to *matchability*. A registered `block` must be a *positive* assertion's phrase — an `assertNotIn` cannot be mutation-proven, since blanking cannot restore an absence — so pair each negative guard with a positive framing assert carrying the entry. Run skills guards via `discover -s skills/tests`: the harness does a bare `import mutation_engine`, so a dotted-name run dies ModuleNotFoundError.
- 2026-07-16 (M56+M65, consolidated M78/M83): "green" is only as wide as what you ran. Bash's working directory persists across tool calls and piping a verify through `| tail -1` makes the chain exit-blind (`tail` exits 0 even when `discover` ImportErrors) — a lingering `cd cairn` silently skipped all three suites, and at the M65 gate the pipe hid a red exit and the chain shipped it. Run suites from the repo root, check each exit code explicitly, and never chain a commit/push/marker write after a piped run. And `cairn_validate` green is NOT the suites green: a `cairn/references/` edit passing every validate check can still break a `skills/tests` guard, and a trivial-tier docs commit pushed main red because only validate was run — any references or template edit runs the three suites too, trivial tier or not.
- 2026-07-16 (M57, extended M79/M81): a hard validate CHECK parsing human-authored markdown must tolerate cosmetic decoration on the semantic token — the INDEX-line regex double-FAILed correct backticked/`[name](name)` entries; D-023's no-false-positive doctrine binds CHECK parsers, not just scan heuristics. Decoration is only ONE axis of freedom, and fixtures varying it alone pass **vacuously** on PHRASING and LAYOUT (F1/85, F2/95, F3/92, F4/90). Vary every axis the prose is free in, and vary it WHERE THE VALUE UNDER TEST LIVES: M81's 36-cell product wrapped at a split point leaving the date before the cut in every phrasing, so no cell wrapped where it changes the ANSWER, and its `page()` fixture always put the block LAST, making trailing-paragraph absorption untestable. Widening a capture class admits non-targets too (F5/85).
- 2026-07-16 (M58+M59+M62, consolidated M77): a repo-wide sweep and a grep-shaped AC both hit ground you did not mean. An exclusion list may name only history files (DECISIONS/CHANGELOG/legacy/reviews-archive), never a live directory — excluding `cairn/references/` left two live citable pages asserting the doctrine still lived in tracking-rules (F1/85; extends M48). And an AC whose evidence is a grep will hit the milestone's OWN artifacts: the guard's `assertNotIn` (an absence-assert is a hit for the token it locks), plus the milestone file's Scope/Tasks and the ROADMAP lineage row on a removal/rename sweep. - 2026-07-16 (M58+M59+M64+M65, consolidated M78/M82): a mutation-registered `block` must sit on ONE physical line, occur exactly once, and include its trailing punctuation — a reflow, a duplicate, or a period→semicolon edit all "found 0"-error the harness. It fires hardest on BRAND-NEW prose authored wrapped then registered (M78 hit it three times in one milestone despite this lesson existing), and just as hard on REFLOWING existing prose: re-wrapping to accommodate a neighbouring edit split a registered phrase across two lines and reddened the suite. Author the anchor on its own physical line, or pick one already unique and unwrapped; re-run the skills suite after any rewording near a registered block; fix the WRAP, never the assert — loosening an assert is how a label→rule guard stops pinning its rule. Also: prose naming a validate finding uses the emitted label verbatim in backticks (`coverage complete`); write evidence counts from command output, never memory (M28).
- 2026-07-16 (M60): refines M19 — hook *registrations* (hooks.json) snapshot at process start, but each script BODY is re-read per invocation from the symlinked checkout: editing an already-registered hook goes live mid-session (the new merge_guard renamed the marker during M60's own merge), while a newly-registered hook stays dead until a fresh conversation; a fixture test still proves only what a hook prints, and live-fire plans must split the two cases.
- 2026-07-16 (M60, extended M80/M85): a SUBSTRING anchor gives false coverage — and prose you add can create one for a guard in ANOTHER file. Shipping a name containing an asserted name (merge_guard → merge_guard_post) unfalsifies every bare `assertIn` on the shorter name AND blocks its mutation registration — use word-bounded `assertRegex` (`\bname\b`; `_` is a word char) (F1/90). On a template, `extraction:` occurred in the file's OWN comment header and in the real field, so deleting the field left the guard green (F3/65). M85 hit the REVERSE: new header prose naming the verb set added a second `unverified`, degrading a bare `assertIn` elsewhere (F1/93) — after adding prose, grep every word an existing guard anchors on. - 2026-07-16 (M61): the mutation harness runs a guard as a SINGLE method (`TestCase(method).run()`), which skips `setUpClass` — a guard that caches its source text in setUpClass reads the unmutated file and reports false coverage on itself; read the target per-test (property/setUp), never a class-level cache.
- 2026-07-16 (M61): don't bulk-edit tracking files with ad-hoc string scripts — a review-time `str.partition` tick-the-boxes script silently truncated the milestone file at the section heading (only `git checkout --` recovered it); tick checkboxes with targeted Edit replacements, and if a script must touch a tracking file, `wc -l`/diff it before moving on.

- 2026-07-18 (M71, corrected M75): a Claude Code hook `matcher` made only of `[a-zA-Z0-9_|]` (or `[a-zA-Z0-9_|, -]`) takes a LITERAL path — split on `|`/`,`, each alternative exact-matched — so `|` does NOT buy regex treatment; only a metacharacter (`mcp__.*__spawn_task`) reaches `new RegExp`, and that path is unanchored. - 2026-07-18 (M72): extends M33 — merge_guard's containment check fires on any Bash command that merely *quotes* the guarded string, not just one that chains it: a heredoc documenting the merge command, an evidence grep, and a debug probe were each denied though nothing merged. - 2026-07-18 (M74, extended M76/M86): when guarded prose maps a LABEL to a RULE or to a SET, the assert must pin the LABEL together with its members on ONE physical line. The harness catches neither — blanking is not swapping, and registration is per file, so sound entries elsewhere mask it. - 2026-07-18 (M81, extended M82): when a change to a SHARED parser is defended with a safety claim, check the claim answers the right question — "a more generous block means no page that passed can now fail" was true and was the WRONG invariant, because the hard CHECK it feeds asks only EXISTENCE questions, so a wider read cannot create failures, it ERASES them. - 2026-07-18 (M83): negation is a property of a CLAUSE, not a phrase list — matching affirmative verbs against a fixed set of negative PHRASES broke in both directions at once when the verb set grew (`checked against`/`read against` were added but only `verified` had a negative form), so `no claim checked against the source` read as an affirmative verification: a false positive on the very prose that motivated the milestone AND a clean bill for a page saying in plain words it was never checked. When a matcher gains verbs its negation handling gains them too; scope the negator search to the verb's own clause, or a trailing `not re-read since` sweeps up the page it merely qualifies.
- 2026-07-18 (M84): a guard whose only assertion is an `assertNotIn` is vacuous against a CRASH — empty stdout satisfies every absence-assert, so a test named for the absent-file path stayed green while the advisory raised on every run and validate exited 1 (F2/90, reproduced independently by two agents). Pair an absence-assert with a POSITIVE signal that the path ran (`OK <label>` in the output), and check what the exit code SHOULD be — a deleted scaffold file makes it 1 for an unrelated reason, so asserting 0 would have been wrong too.
